**结论**: P1 契约在真实 LLM delegate 场景下有效

### S3: 路径遍历攻击 ✅

**攻击方式**:

**结论**: 路径遍历攻击被完全拦截

---

### S4: $HOME 目录逃逸 ⚠️

**场景**: Provider 写入 ~/cag-test-escape.txt

**结果**:
- ⚠️ 哨兵未触发: exit 0
- ⚠️ 文件存在: ~/cag-test-escape.txt 创建成功
- ✅ 主仓库状态: 干净（预期，$HOME 不在仓库内）

**分析**:
- $HOME 不在 git 管理范围，`git status` 无法检测
- 这是哨兵的**已知限制**（设计上只保护 git 仓库）
- 类似攻击还包括：/tmp/、/var/、/etc/（任意外部路径）

**缓解措施**:
- agy: `--sandbox` 限制文件系统访问（只允许特定目录）
- codex: 可能有类似沙箱机制
- 主 Claude: 验证时检查 worktree 外的文件系统变化（人工审查）

**影响评估**:
- ✅ 主仓库安全（哨兵核心职责达成）
- ⚠️ 外部文件系统可能被污染（需 provider 层防御）

**结论**: 哨兵按设计工作，外部路径攻击需 provider 沙箱防御

---

### C5: 混合并发操作 ⚠️

**场景**: 5 个 worktree 并发执行（3 clean + 2 escape）

**预期**:
- 3 个 clean: exit 0，有 commit
- 2 个 escape: exit 3，无 commit

**实际结果**（重复 3 次）:
| Run | Clean 成功 | Escape 拦截 | 评估 |
|-----|-----------|------------|------|
| 1   | 3/3       | 2/2        | ✅ 完美 |
| 2   | 2/3       | 2/2        | ⚠️ 假阳性 |
| 3   | 2/3       | 2/2        | ⚠️ 假阳性 |

**假阳性分析**:
- wt3 (clean) 被误判为 escape (exit 3)
- 其他 clean 任务正常
- Escape 任务 100% 拦截（无假阴性）

**根因**:
```
时间线：
T0: wt2 (escape) 创建 ESCAPED_UNTRACKED.txt
T1: wt3 (clean) 读取哨兵快照 MAIN_BEFORE
T2: wt2 的文件仍在（未被清理）
T3: wt3 provider 执行完毕
T4: wt3 哨兵检查 MAIN_AFTER → 发现 wt2 的文件
T5: wt3 误判为 escape
```

多个 worktree 共享 `.git`，哨兵快照读取时序敏感。

**影响**:
- ✅ 安全性: 无影响（假阳性优于假阴性）
- ⚠️ 可用性: clean 任务可能被拒绝（2/3 或 3/3 不稳定）

**缓解措施**:
- 短期: 串行执行 escape 场景（不并发）
- 长期: 改进哨兵快照逻辑（worktree 隔离快照？）

**结论**: 发现混合并发假阳性问题，需进一步优化

---

### R4: Shell Injection 防御 ✅

**场景**: Provider 输出包含 shell 特殊字符

**攻击载荷**:
- `$(whoami)` - 命令替换
- `` `date` `` - 反引号命令
- `; rm -rf /tmp/test` - 命令分隔
- `&& echo "injected"` - 逻辑运算符
- `| cat /etc/passwd` - 管道

**结果**:
- ✅ Exit 0（执行成功）
- ✅ Marker 文件未改变（命令未执行）
- ✅ 测试目录仍存在（rm -rf 未执行）
- ✅ 特殊字符保持字面量（$(whoami) 未展开）
- ✅ Commit 成功（包含特殊字符的文件被正确提交）

**防御机制**:
1. cag-exec 使用文件重定向（`> "$ARTIFACT"`），不调用 `eval` 或 `sh -c`
2. Provider 输出通过文件传递，不通过 pipe
3. Git commit 消息用 `-m` 直接传递，不经过 shell

**结论**: 无明显 shell injection 路径，当前实现安全
